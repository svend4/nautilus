## 1. Базовый принцип — «не слияние, а совместимость»

### Метафора Office Suite

> Как Office Suite читает `.docx`, `.pdf`, `.xlsx` не сливая их в один формат —
> Nautilus читает все репозитории экосистемы, находит связи, строит общий вид.

Это фундаментальный дизайн-принцип: портал **не унифицирует** форматы, а строит над ними мета-слой связей. Каждый репозиторий сохраняет:

- свой нативный формат (`.info1`, `.pro2`, `.meta`)
- свою атомарную единицу (документ, концепт, правило)
- свою внутреннюю логику

Nautilus добавляет:

- общий интерфейс чтения (`BaseAdapter`)
- общее пространство координат (Q6)
- общий язык связей (`PortalEntry.links`)
- общий критерий истины (консенсус)

### Почему это важно концептуально

Альтернативой был бы **монорепозиторий** или **единый формат**, что привело бы к потере специализации каждого источника. Nautilus выбирает путь **федерации с протоколом**, а не **централизации с форматом**.

## 2. Лестница совместимости (4 уровня)

### Формальное определение

| Уровень | Название | Семантика | Требования |
|---------|----------|-----------|------------|
| **0** | обнаруживаемый | портал знает о существовании репо | запись в `nautilus.json` |
| **1** | читаемый | возвращает список записей | статический адаптер |
| **2** | связанный | Q6-координаты + кросс-ссылки | `metadata.q6` + `links` заполнены |
| **3** | интерактивный | живой поиск через GitHub API | реальный `fetch()` с запросом |

### Распределение в текущей экосистеме

| Уровень | Количество | Репозитории |
|---------|------------|-------------|
| 3 | 1 | pro2 |
| 2 | 3 | data7, infosystems, ai_agents |
| 1 | 3 | info1, meta, data2 |
| 0 | 0 | — |

**Наблюдение:** ни один репо не остался на L0. Все преодолели минимальный порог читаемости.

### Концептуальный смысл лестницы

Это **миграционная модель зрелости**. Репо может подключиться за 5 минут на L1 и постепенно дорастать до L3 без переписывания портала. Каждый уровень — это договор (`contract`) между репо и порталом.

## 3. Q6-пространство (6-битная онтология)

### Структура

- **6 бит** = 64 ячейки = 64 вершины гиперкуба = 64 гексаграммы И-Цзин
- **Метрика:** Hamming-расстояние (0–6)
- **Универсальность:** любой концепт любого формата получает адрес `[b0..b5]`

### Ключи отображения
### Ключи отображения (`passport_schema.json:42`)

Каждый формат определяет правило проекции своих сущностей на Q6:

| Репо | Правило |
|------|---------|
| info1 | `alpha+4` → 3 старших бита |
| pro2 | нативные Q6-координаты |
| meta | `hex_id - 1 → bin(6)` |
| data7 | `порядковый номер % 64 → bin(6)` |

### Hamming-соседство как семантическая близость

Концепты на расстоянии 1 в Q6 — «семантически соседние». Это обеспечивает:

- Поиск «похожего» без ML-эмбеддингов (`q6_neighbors`)
- Кластеризацию (`cluster.py`, radius=1 → 12 кластеров)
- Обнаружение пробелов (`gap_detection.py`)
- Heatmap покрытия (`q6_map.html` — режим тепловой карты)

### Текущее Q6-покрытие

```
Всего вершин:    64
Покрыто (real):  14  (21.9%)
Слабые (fb):     19  (29.7%)
Пробелы:         31  (48.4%)  ← главный концептуальный вызов

CA-класс II (периодические): только 8/50 покрыто (16%) — критично
```
- Поиск «похожего» без ML-эмбеддингов
- Кластеризацию (`cluster.py`)
- Обнаружение пробелов (`gap_detection.py`)
- Heatmap покрытия (`q6_map.html`)

### Философское содержание

Q6 — попытка построить **минимальное достаточное пространство** для онтологии знаний. 6 бит — это компромисс между:

- Мнемоническим пределом Миллера (7±2)
- Бинарной природой цифрового представления
- Мощностью 2⁶=64, совпадающей с древними классификациями (И-Цзин, таро)

## 4. Консенсус как онтологический критерий

### Формальное определение

```json
"consensus_threshold": 1.0
```

Концепт признаётся «согласованным» (`agreed: true`) **только если найден во всех адаптерах** экосистемы.

### Три типа покрытия

| Поле | Смысл |
|------|-------|
| `coverage` | доля адаптеров с реальными записями |
| `coverage_with_fallback` | доля адаптеров с любыми записями (вкл. заглушки) |
| `agreed` | булев флаг полного консенсуса |

### Философское утверждение

> Истина = пересечение независимых формализмов.

Если концепт виден только в одном репо — это **гипотеза**. Если в двух — **вероятная закономерность**. Если во всех — **согласованный факт экосистемы**.

### Частичный консенсус как инструмент

Вывод `present_in: [...], missing_in: [...]` позволяет определить:
Это не просто метрика — это **эпистемологическая позиция**:

> Истина = пересечение независимых формализмов.

Если концепт виден только в одном репо, это **гипотеза**. Если в двух — **вероятная закономерность**. Если во всех — **согласованный факт экосистемы**.

### Частичный консенсус как инструмент

Вывод `present_in: [...], missing_in: [...]` позволяет точно определить:

- В каких формализациях концепт существует
- Где его нет (возможно, требует добавления)
- Где только заглушка (требует реального содержания)

## 6. Пять путей онбординга

### Миграционная модель

| Вариант | Подход | Автоматизация | Время |
|---------|--------|---------------|-------|
| **A** | ручной адаптер | 0% | ~20 мин |
| **B** | `generate_passport.py` мастер | ~50% | ~5 мин |
| **C** | `nautilus.json` в целевом репо | ~80% | ~10 мин |
| **D** | `scan_repo.py` авто-сканер | ~95% | ~2 мин |
| **E** | GitHub Actions webhook | 100% | ~30 мин настройки |

A → E — это эволюция от **pull-модели** (портал тянет данные) к **push-модели** (репо сообщает о себе).

## 8. Мостовая семантика (bridges)

### Текущий статус

- ✅ Мосты объявлены в каждом паспорте (7 паспортов)
### Концептуальная прогрессия

A → E — это эволюция отношений портал↔репо:

1. **A:** портал сам интегрирует репо
2. **B:** портал помогает интегрировать
3. **C:** репо объявляет себя совместимым (`nautilus.json` в корне)
4. **D:** портал сам обнаруживает и описывает репо
5. **E:** репо уведомляет портал о себе через CI

Это переход от **pull-модели** (портал тянет) к **push-модели** (репо сообщает).

## 7. Мостовая семантика (bridges)

### Формат объявления

```json
"bridges": {
  "pro2":  "alpha-уровень ↔ глубина Q6-концепта",
  "meta":  "онтология ↔ гексаграммы класса I",
  "data7": "bidir_train ↔ K₀→K₁→K₂ цикл"
}
```

### Что это

Мосты — **типизированные семантические отображения** между форматами. Это не просто ссылки — это декларация того, **как именно** два формализма связаны.

### Текущий статус

- ✅ Мосты объявлены в каждом паспорте
- 🟡 Пока выражены на естественном языке
- ❌ Нет формальной алгебры мостов (композиция, обратимость, транзитивность)

Это **открытый концептуальный вопрос** для v2.0.
